# Square Dispute (Chargeback) Evidence — Runbook & API Notes

How we respond to Square chargebacks for **deposit-based** reservation disputes, and the
Square Disputes API gotchas we learned responding to them. First live submission:
dispute `cekOSvWYFueQDzGHkfPRoC` (RACE03928151, $136.26, reason `DUPLICATE`) — submitted
2026-06-30, state moved `EVIDENCE_REQUIRED → PROCESSING`.

> **Scope guardrail — deposit disputes only.** Automation should act only when the disputed
> payment is a reservation **deposit** (the payment maps to a
> `bowling_reservations.square_deposit_payment_id` row; the Square order note begins `Deposit -`).
> Disputes on group-function balances, day-of charges, gift-card sales, or anything not tied to a
> deposit row must be **skipped and routed to manual review** — the standard narrative ("a single
> deposit funded the service; the card was charged once") is only true for deposits.

## The evidence packet

A strong packet for a deposit dispute is **short text recaps + detailed PDFs**, each mapped to a
Square `DisputeEvidenceType`:

| Kind | evidence_type                          | Content                                                                      |
| ---- | -------------------------------------- | ---------------------------------------------------------------------------- |
| TEXT | `REBUTTAL_EXPLANATION`                 | ≤500-char recap of the merchant's position                                   |
| TEXT | `GENERIC_EVIDENCE`                     | ≤500-char "not a duplicate / single charge" recap                            |
| PDF  | `REBUTTAL_EXPLANATION`                 | Full rebuttal statement & case summary                                       |
| PDF  | `RECEIPT`                              | Itemized receipt + booking itemization                                       |
| PDF  | `CANCELLATION_OR_REFUND_DOCUMENTATION` | Clickwrap acceptance log (ts, IP, UA, policy version) + verbatim policy text |
| PDF  | `SERVICE_RECEIVED_DOCUMENTATION`       | Proof the reservation was fulfilled (status, race date)                      |
| PDF  | `DUPLICATE_CHARGE_DOCUMENTATION`       | Single-charge proof — required as a FILE for DUPLICATE-reason disputes       |

The acceptance evidence comes from the `clickwrap_acceptances` table (written by
`recordClickwrap` / `POST /api/clickwrap/record`): timestamp, server-captured IP + user-agent,
`policy_version`, email, phone, name, bill_id, amount, card last-4/brand. That row is the single
strongest piece — it ties the cardholder's identity and device to the all-sales-final policy at
the moment of payment.

PDFs are generated by rendering print-styled HTML through headless Chrome:

```
chrome.exe --headless --disable-gpu --no-pdf-header-footer \
  --user-data-dir=<unique temp dir> \
  --print-to-pdf=<out.pdf> file:///<page.html>
```

## Square Disputes API gotchas (learned the hard way)

1. **`evidence_text` is capped at 500 characters.** Long-form narrative must go in a PDF; the text
   endpoint is for short recaps that point at the files.
2. **Some evidence types are file-only.** `DUPLICATE_CHARGE_DOCUMENTATION` rejects text with
   `400 BAD_REQUEST` ("Use CreateDisputeEvidenceFile for this evidence_type"). For a DUPLICATE
   reason, put the recap under `GENERIC_EVIDENCE` text **and** attach a real
   `DUPLICATE_CHARGE_DOCUMENTATION` **PDF**.
3. **Idempotency does NOT return the existing resource.** Re-posting an evidence item with a
   previously-used `idempotency_key` returns `400 IDEMPOTENCY_KEY_REUSED` (it echoes the existing
   evidence id). So a retry that re-uploads already-attached items will "fail" on those. Robust
   pattern:
   - Upload all items; treat `IDEMPOTENCY_KEY_REUSED` as **already-present**, not a failure.
   - `GET /v2/disputes/{id}/evidence` and verify every expected evidence id is present.
   - Only then `POST /v2/disputes/{id}/submit-evidence` — **one call, irreversible.**
4. **File upload is multipart/form-data:** a `request` part as JSON
   (`{idempotency_key, evidence_type, content_type:"application/pdf"}`) plus a `file` part. Don't set
   `Content-Type` by hand — let the HTTP client set the multipart boundary. Files ≤ 5 MB;
   PDF / PNG / JPEG / TIFF / HEIF accepted.
5. **The API keys off the dispute id, not the payment id.** The Square receipt URL exposes the
   _payment_ id (`…/receipt/preview/{paymentId}`). Resolve the dispute from the payment via
   `ListDisputes` → `disputed_payment.payment_id`. Automation input should accept a payment id and
   resolve the dispute id.

## When can evidence be uploaded?

| Dispute state                            | Upload evidence? | Notes                                                             |
| ---------------------------------------- | ---------------- | ----------------------------------------------------------------- |
| `INQUIRY_EVIDENCE_REQUIRED`              | ✅               | Pre-formal inquiry stage                                          |
| `EVIDENCE_REQUIRED`                      | ✅               | Main window — add **and** remove (`DeleteDisputeEvidence`) freely |
| `PROCESSING`                             | ❌               | Already submitted; locked, no add/change/remove                   |
| `WON` / `LOST` / `ACCEPTED` / `*_CLOSED` | ❌               | Terminal                                                          |

`submit-evidence` is the one-way door: it flips the dispute to `PROCESSING` and locks all evidence.
Nothing reaches the card network until that call.

## Recommended production flow

1. Resolve dispute id from the payment id; confirm `state` is `EVIDENCE_REQUIRED`.
2. **Deposit gate** — confirm the disputed payment maps to a deposit row; else skip to manual.
3. Gather verified facts from Neon (`bowling_reservations`, `clickwrap_acceptances`) + Square.
4. Generate the PDF packet + text recaps.
5. Upload all items → `GET …/evidence` to verify → **hold**.
6. Submit. Recommended: **auto-upload but hold `submit-evidence` for explicit confirm**, or
   auto-submit only inside the last 24h before `due_at`, so there's a review window while evidence
   is still unlocked without risking the deadline.
